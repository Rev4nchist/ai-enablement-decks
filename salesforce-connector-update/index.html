What this connector actually is

Salesforce now offers its own AI-ready data servers. We could let Claude talk to them directly — but then the company would have little control over who connects, what they can do, or what they accessed.

Our connector sits in the middle as a governed doorway. Claude talks to our connector; our connector talks to Salesforce. That middle position is where the company gets its control: it decides who is allowed in (using the corporate Microsoft directory), restricts everyone to read-only access, only shows the handful of tools each role actually needs, and keeps a durable record of every action. The end user just asks a question and gets an answer — all the governance happens invisibly behind the scenes.

The second reason is just as important: that same middle position lets us enhance and customise the experience for Fourth. Because every request flows through our connector (built on the FastMCP framework), we're not limited to passing raw Salesforce data straight through — we can shape and improve the connection in ways the off-the-shelf Salesforce connector cannot:

📊
Interactive results, not walls of textAnswers can render as sortable, searchable tables, charts and dashboards inside the Claude conversation.
✨
Visualizations built on the flyThe AI can construct a custom dashboard or chart tailored to the exact question, not a fixed template.
⚙️
Fourth-specific behaviour, added onceCustom rules — caching, formatting, department-aware routing — layered in centrally so every tool inherits them.
💬
The connector can ask clarifying questionsIf a request is ambiguous (which region? which period?), it can pause and ask rather than guess.

In short, the proxy gives us a place to make the Salesforce experience smarter and more Fourth-specific over time — not just safer.

The story, in order

Friday 29 MayBreakthrough

The session with Ivan. Two things were built but never proven for real: (1) can a Microsoft login be handed to Salesforce automatically, and (2) can Salesforce's new AI data servers serve data back through our connector? We worked through it one obstacle at a time:

First obstacle — "user hasn't approved this consumer." The identity hand-off reached Salesforce and it correctly recognised the employee, but then refused because its approval policy was blocking automated sign-in. Ivan adjusted the policy and granted the right permission set → success.

Second obstacle — the data gateway said "invalid token." Salesforce accepted the identity but wouldn't serve data. We researched it and found two specific Salesforce settings were required (a dedicated access scope, and an option to issue the right type of token for named users). Ivan switched both on → the gateway returned real data.

Outcome: every external dependency closed in that one session. The project moved from "we don't know if this is even possible" to "it works — now it's just our code to finish."

A useful side note: Anthropic's own built-in Salesforce connector was broken at the time by a bug on their side. Our governed proxy steps around that bug — so for now ours is the working path from Claude to Salesforce, and it's the governed one.
Sunday–Monday 1 JuneBuild

Wiring the live data connection. Connected our code to Salesforce's data gateway (which needs a short multi-step "handshake" before it will answer — handled automatically), added the read tools, added a check that blocks unsafe or malicious data requests before they reach Salesforce, and proved real account data flowing through the production path.

Tuesday 2 JuneGo-live

The sign-in switch, and go-live. Our first login method ("Easy Auth") worked in a browser but could not drive Claude's connector — Microsoft instantly rejected the request because of an extra parameter Claude includes. We switched to a different component ("AzureProvider") that removes that parameter and completes the sign-in normally. A "what could go wrong" review before coding caught a token-storage risk (we added encryption before it shipped). The connector then went LIVE and returned real Salesforce data. We also fixed the data map so the AI only sees the fields a given user can actually read.

Wednesday 3 JuneSecurity review + Alpha

Independent review, redeploy, and Alpha release. The pull request triggered an automated code review (CodeRabbit). It flagged real customer data (company names, revenue numbers) that had been captured into test files during live testing — credential scanners don't catch customer data, so it had slipped through. We replaced it all with fictional placeholders, removed a hardcoded personal ID, deleted seven throwaway diagnostic scripts, and added a fail-fast check so the server refuses to start if misconfigured. Re-reviewed by a second, independent AI from a different vendor, then redeployed as revision 0000009 — verified live.

Alpha release to a limited team. With the reviewed version deployed, the connector was opened to a small alpha group for hands-on use: Jeff, Jen, Carly, CB, Borja, Mark, Danny, and Ivan. A deliberately small first audience — the goal is real-world feedback before access is widened.

Live re-evaluation (later on 3 June) — good news. A fresh end-to-end test confirmed the alpha is in better shape than the morning's notes suggested: Opportunity and Case are queryable now (pipeline, amounts, cases all return real data — earlier we'd flagged these as "waiting on a Salesforce-side feed", now resolved for querying), and relationship traversal works (a real account returned 302 linked opportunities and 339 linked contacts). One narrower gap remains, and it's ours to fix, not Ivan's: the relationship-discovery helper (which lists related record types) doesn't yet surface Opportunity/Case, and the object catalog shows stale field counts for them. The next phase switches schema discovery to Salesforce's native "describe" service, which fixes the gap and removes the earlier dependency on Ivan's separate feed.

The four things worth understanding

🔗 The identity bridge

One Microsoft sign-in is automatically translated into Salesforce access. No separate Salesforce login, no API keys. The connector asks Microsoft for permission to act on that user's behalf, hands the proof to Salesforce, and Salesforce maps it to the matching employee.

End userSign in once with the account you already have. Everything else is invisible.
Security & oversightEvery action ties to a named person. The sensitive credential lives per-user in the corporate secrets vault — never in code.

🔐 The sign-in switch

We changed the login component so Claude's connector can actually sign in. The new approach is fail-closed: a caller gets in only with the right corporate role and through the approved application. Anything else is refused.

End userIf you have the right role, it just works; if not, you simply can't connect.
Security & oversightNo role, no token, or wrong app = immediate denial. No partial or accidental access.

🛡️ Governance middleware

Every request passes a series of checkpoints: Audit (records who/what/when, never the raw query), Allowlist (only role-appropriate tools), Safety (independently blocks any change to data and oversized queries), and Rate limiting (protects the public login endpoints).

End userInvisible in normal use; negligible delay.
Security & oversightNo usage without an audit trail. Read-only is enforced in three independent places.

🔍 The security reviews

Three independent layers, not one: an internal premortem before coding (caught the token-encryption risk early), a cross-model adversarial review (a second AI from a different vendor), and the automated PR review (caught the real customer data in test files).

End userA more reliable, better-tested product.
Security & oversightIssues caught at three stages — cheapest and safest when caught early.
