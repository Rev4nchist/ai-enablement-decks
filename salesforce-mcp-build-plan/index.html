fourth-salesforce-mcp — Build & Deployment Plan

FastMCP v3 governance proxy on Azure Container Apps, wrapping Salesforce's Hosted MCP. Entra-rooted identity · audit · rate limit · tool curation · custom-object schema awareness.
Draft for Dave's review Scaffold ✅ done (commit fefb034) ⏳ Awaiting Entra v2 issuer flip v1 → v2 framing · 2026-05-07
One-line summary: Salesforce now ships official Hosted MCP servers (TDX 2026). Our job is no longer building a Salesforce wrapper — it's building the governance proxy that sits in front of theirs. Entra-aligned auth, audit, rate limit, dept routing, custom-object schema. Phase 1 ships a Marketing read-only pilot.

§1 Context — why this is being built

Fourth needs a per-user Salesforce surface for Claude/MCP clients with Entra-rooted identity, audit trail, rate limiting, department-specific tool curation, and custom-object awareness. The original plan (March 2026) was to hand-roll a Salesforce REST/SOQL wrapper.

Reframed 2026-04-17. Salesforce announced Headless 360 / Hosted MCP Servers at TDX 2026 (Apr 15) — first-party MCP endpoints covering our entire Phase 1 scope (sObject CRUD, SOQL/SOSL, Apex Invocable, Flows, Named Queries, Data 360, Tableau Next, Prompt Builder, DevOps Center).

The build is now a governance proxy in front of Salesforce's Hosted MCP. Salesforce owns tool plumbing; Fourth owns identity brokering, audit, rate limiting, tool curation, schema intelligence, and protocol insulation.

Phase 1 Definition of Done

A Marketing user asks Claude a Salesforce question → Claude routes through salesforce-mcp → the proxy authenticates that user via Entra → the proxy proxies to Salesforce Hosted MCP under that user's permissions → the answer respects field-level security and sharing rules → the audit log proves what happened.

§2 Recommendation summary

Build fourth-salesforce-mcp as a FastMCP v3 governance proxy on Azure Container Apps, wrapping Salesforce's Hosted MCP servers.

  • Transport: Streamable HTTP with stateless_http=True
  • Edge auth: Easy Auth + Return401 + EasyAuthGate middleware (Entra)
  • Downstream auth: OAuth 2.0 + PKCE with a Salesforce External Client App; OBO / token exchange bridges Entra → Salesforce
  • First persona: Marketing read-only bundle (~10 curated tools)
  • Inheritance: Project shape from gong-mcp, Azure deploy from the playbook verbatim

§3 Strategic decisions

With rationale and sources. Reject any of these in code review and you've reopened the architecture debate.

D1Governance proxy, not REST wrapper
Salesforce now owns tool plumbing — duplicating it surrenders speed without buying control. Proxy preserves Fourth's value (Entra, audit, dept routing, tool curation).
Source: Parent project Notion doc, 2026-04-17 architecture reframe
D2FastMCP v3 with [apps] extra
70%+ of MCP servers; middleware lifecycle covers our governance needs; v3 ships native FastMCP.as_proxy() for upstream MCP wrapping.
Source: playbook §Stack; gofastmcp.com/servers/proxy
D3Streamable HTTP + stateless_http=True (no sticky sessions)
Microsoft + FastMCP 2026 guidance: stateless is preferred for Streamable HTTP. Sticky sessions force Single-revision mode and block blue-green.
Source: playbook anti-patterns; gofastmcp.com/deployment/asgi; learn.microsoft.com/azure/container-apps/mcp-overview
D4Edge auth: Easy Auth + Return401 + EasyAuthGate
Microsoft's documented pattern for MCP-on-ACA. RedirectToLoginPage breaks MCP clients (no HTML redirect). AllowAnonymous lets unauthenticated traffic through. EasyAuthGate enforces App Role as defense-in-depth.
Source: playbook Phase 6+8; learn.microsoft.com/azure/container-apps/mcp-authentication
D5Downstream: External Client App + OAuth 2.0 + PKCE; OBO/token-exchange
Salesforce Hosted MCP requires an External Client App (legacy Connected Apps not supported); PKCE S256 mandatory; no DCR. OBO (RFC 8693) is the 2026 best practice for Easy-Auth-fronted servers needing a downstream OAuth token.
Source: help.salesforce.com External Client Apps for MCP; developer.salesforce.com hosted-mcp; learn.microsoft.com OBO flow
D6Tool curation via facade pattern
Salesforce's own d360-mcp-server fronts 187 operations behind 3 semantic tools (search, payload_examples, execute). MCP clients degrade beyond ~50 tools. Phase 1 ships marketing-readonly bundle of ~10 curated tools.
Source: github.com/forcedotcom/d360-mcp-server; Notion onboarding guide §5
D7Custom object schema map (one-time + cached)
Fourth has many custom objects — Claude will hallucinate field names without a schema source of truth. Build via discovery script, cache in Cosmos schema_cache, expose via salesforce_get_schema_summary.
Source: Notion onboarding guide §6 + §13
D8Azure footprint matches playbook exactly
ca-mcp-salesforce in locmap-env (Sweden Central), shared agentarchacr, shared agent-arch-kv-prod, dedicated Cosmos DB mcp-salesforce, SystemAssigned MI, dual-tag (semver + sha-short), deploy by digest.
Source: playbook §Naming + §Phase 8
D9Inherit gong-mcp shape verbatim
Closest playbook-aligned reference. Reuse: config.py env dataclass, rate_limit.py, Dockerfile, conftest.py env-isolation, smoke tests, Railway lessons. Do NOT inherit gong-mcp's gaps (no auth.py, no workflows, Procfile in COPY).
Source: scout audit of gong-mcp

Phase 2 · Track A in progress

Hosted MCP ProxyClient

Implement salesforce_mcp/hosted_mcp/client.py using FastMCP.as_proxy() over httpx pointed at Salesforce Hosted MCP. Watch the oracle-flagged gotchas: (a) auth header forwarding requires explicit pass-through config, (b) no built-in token caching — wrap with our own.

Until Phase 0's URL is confirmed: stub with a fixture returning a hardcoded tool list matching forcedotcom/mcp-hosted wiki. Tests use pytest-httpx to mock the endpoint.

Phase 3 · the hard part

Auth — edge + downstream

  1. Edge auth (Entra → our proxy): EasyAuthGate middleware verbatim from playbook §Phase 6. Tests: missing header denies; bad base64 denies; principal without Mcp.User denies; both claim forms recognized.
  2. Downstream auth (proxy → Salesforce): OBO token exchange. Caller's Entra token → MS /v2.0/token OBO → returns Entra token with api://<sf-app-id>/.default audience → bearer to Salesforce ECA → ECA mints SF access + refresh tokens. Cache RTs in KV; access tokens in-memory with TTL.
Open question: Salesforce ECA may not accept an Entra-issued token directly even with OBO. May need SAML SSO bridge instead. Confirm in Phase 0 with Salesforce docs / AE.
Phase 4 · Track C

Schema discovery

scripts/discover_schema.py --org <sandbox|prod> --output data/schema/salesforce-schema-map.json. Hits services/data/v60.0/sobjects + per-object describe + child-relationship walk.

Output: salesforce-schema-map.json (full structured map) + salesforce-schema-summary.md (Claude-readable). Exposed via salesforce_get_schema_summary and salesforce_describe_object. Cached in Cosmos schema_cache (PK /org_id), refreshed weekly.

Critical: capture custom-object patterns (__c, __r, __mdt, __x, __b) and FLS per current user. Without this, Claude hallucinates field names against Fourth's custom objects.

Phase 5 · Track B

Governance middleware

Order matters:

Request ID -> Error handling -> EasyAuthGate -> Audit start
  -> Department routing -> Tool allowlist -> Safety -> Rate limit
  -> ProxyClient -> Audit complete

Phase 1 allowlist · marketing-readonly

allow:
  - salesforce_list_available_objects
  - salesforce_describe_object
  - salesforce_get_schema_summary
  - salesforce_query_readonly         # SOQL, default LIMIT 25, max 200
  - salesforce_search_readonly        # SOSL
  - salesforce_get_record
  - salesforce_get_related_records
  - salesforce_explain_object_relationships
  - salesforce_refresh_schema_cache
  - salesforce_list_allowed_tools
deny:
  - any_mutation · any_delete · execute_flow
  - invoke_apex_action · any_data_360_writeback

Safety middleware blocks: mutation tool names, queries against unmapped custom objects, unbounded queries, queries projected to return >max-rows.

§16 Schema discovery — "does this customize the connector for Fourth's custom objects?"

Yes — that is precisely what it does, and it's why v1 includes it. Fourth's Salesforce org has many custom objects (__c, __r, __mdt, __x, __b), each with their own fields, relationships, picklists, validation rules, and FLS policies that don't exist in any other org. Without schema discovery, Claude operates against a "generic Salesforce" mental model and hallucinates field names whenever Fourth's customization diverges from standard.

What the discovery process produces

  1. data/schema/salesforce-schema-map.json — full structured map of every object visible to the discovery user:
    • API name + label + plural label
    • Custom flag, queryable/searchable/createable/updateable/deletable flags
    • Every field: API name, label, type, picklist values, references, FLS for calling user, required, indexed, external-ID
    • Parent + child relationships (e.g., Account → Implementation_Project__c via Account__c)
    • Record types
  2. data/schema/salesforce-schema-summary.md — Claude-optimized markdown index:
    • One section per object
    • Custom objects + fields highlighted (__c suffix called out)
    • Traversable relationships listed under each object
    • Designed to fit comfortably in Claude's working context for a session
  3. Cosmos schema_cache container (PK /org_id) — same data served at runtime via the salesforce_get_schema_summary MCP tool

How it customizes the connector at runtime

  • The proxy ships with salesforce_get_schema_summary and salesforce_describe_object MCP tools registered out of the box
  • System prompt for Salesforce tools tells Claude: "Consult salesforce_get_schema_summary before writing SOQL"
  • When a user asks "what's the Customer Health Score for AcmeCo?", Claude reads the summary, sees Implementation_Project__c.Customer_Health_Score__c and its Account__c reference, and writes correct SOQL:
SELECT Customer_Health_Score__c FROM Implementation_Project__c
 WHERE Account__r.Name = 'AcmeCo' LIMIT 25

…instead of guessing standard-looking field names that don't exist (Account.healthScore, etc.).

Safety middleware (Track B) enforces "describe before query": SOQL against an unknown custom object is blocked until Claude confirms it via salesforce_describe_object.

Refresh strategy in v1

  • Initial: python scripts/discover_schema.py --org production runs during the first deploy
  • Periodic: weekly cron (Azure Functions or scheduled Container Apps job) re-runs discovery and refreshes Cosmos
  • Manual: salesforce_refresh_schema_cache MCP tool callable by admin-tier users on demand

What v2 adds on top

  • Per-user FLS views — v1 caches a baseline; v2 re-describes under each calling user's identity at first call so the summary reflects exactly what that user can see
  • Schema diffs over time — audit trail of changes (governance + detecting breaking migrations)
  • Breaking-change alerts — notify when a known field disappears (signals stale schema → tools may break)
