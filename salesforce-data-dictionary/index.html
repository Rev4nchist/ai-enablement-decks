The opportunity

You flagged it exactly: we need a data dictionary so we're all querying the same fields. Today, two people asking "enterprise pipeline by segment" can each pick a different — but equally valid — field, so the numbers don't match. A data dictionary is the agreed list of which field means what, and which one to use, for every business concept. The powerful part: once you bless it, we bake it into the Salesforce connector so the AI steers everyone to the same fields automatically — consistency by construction, not by memory.

Why this isn't already solved

Ivan's Schema Detector
Alerts when a field is added or removed. A smoke alarm — it doesn't say which field to use.
Connector schema awareness
Stops the AI inventing fields. But lists all real fields — not the one blessed field per concept.
Data dictionary (this)
The standardization layer on top: for each concept, the single canonical field + its meaning.

The proposed dictionary

Read each row as: "when someone asks about this concept, query this field." Standard = a real standard Salesforce field, proposed with confidence. Confirm = org-specific or business-defined — needs a quick RevOps check against our live org.

For each concept we'll also record allowed values (picklists), the owner, and any deprecated "do-not-use" fields — so old habits get flagged automatically.

How it becomes self-enforcing

  1. We store the blessed dictionary as one small, version-controlled file — one row per concept.
  2. The connector already briefs the AI on the real schema before every query. We merge the dictionary into that brief.
  3. Every question then resolves to the same field — in a meeting on your phone or building a report. No one memorizes field names.
  4. When Ivan's detector flags a schema change, that triggers a quick review of one row — so the dictionary stays current.
