01

What changed — and why it matters

Until this release, Azure Foundry was primarily useful for model hosting — you could deploy and call models, but running your own agent logic meant managing your own infrastructure. Foundry Hosted Agents changes that. You can now publish containers directly to the Foundry Agent Service, and it runs your code — not just a prompt definition.

Important: This is a ground-up rebuild of the Ignite 2025 preview — it is not a rolling upgrade, it is a cutover. If anyone on your team ran agents on the old preview backend, those need to be inventoried and migrated.

The problem Foundry is solving here is fundamental: traditional compute wasn't designed for agents. Containers, web apps, and serverless functions were built for web APIs where multiple users share the same instance. That's fine for a REST endpoint, but unsafe for agents — which write files, execute code, and hold sensitive context mid-session. Sharing a container across sessions isn't just inefficient, it's a security nightmare.

02

How it works

The architecture is clean. You write agent code in any framework, wrap it with a lightweight hosting adapter that handles HTTP, health checks, and OpenTelemetry, containerise it, push to Azure Container Registry, and deploy. Foundry handles everything else.

01

Write your agent code

Use any framework — Microsoft Agent Framework, LangGraph, CrewAI, OpenAI Agents SDK, Claude Agent SDK, or plain custom code. Wrap it with the hosting adapter (azure-ai-agentserver-*) which auto-exposes it as an HTTP service with Foundry-compatible protocols.

02

Test locally

The hosting adapter runs a local server on localhost:8088. Test your agent fully with the Foundry Responses API, conversation management, and auth flows before touching a container.

03

Containerise and push to ACR

Write a Dockerfile that gives your agent exactly the tools and dependencies it needs. Build for linux/amd64. Push to Azure Container Registry. Foundry pulls from there at deploy time.

04

Deploy with azd deploy

Azure Developer CLI handles the heavy lifting. Foundry pulls the image, provisions compute, assigns a dedicated Entra ID, and exposes a stable endpoint at https://{project_endpoint}/agents/{agent_name}.

05

Poll for active status

Creating an agent version starts provisioning — it doesn't complete it. Poll the status until it reaches active. Handle creating → active → failed states explicitly in any automation. Don't skip this step.

06

Publish to Teams / M365 (optional)

One-click publish from the Foundry portal. The platform automatically bridges the OpenResponses protocol to Activity Protocol for Teams delivery — no separate wiring. Note: publishing provisions a new dedicated agent identity. Reconfigure RBAC after publishing.

# Your agent environment — total control over deps
FROM python:3.11-slim

WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt

# The hosting adapter handles HTTP, health, OTel
COPY agent.py .
CMD ["python", "agent.py"]
# One command to build, push, and deploy
$ azd deploy

# What happens behind the scenes:
# 1. Packages your code + builds container image
# 2. Pushes image to Azure Container Registry
# 3. Foundry pulls image, provisions compute
# 4. Assigns dedicated Entra ID to agent
# 5. Exposes endpoint with stable URL
# 6. Handles scaling, persistence, observability

Supported Protocols

OpenResponses

Conversational — use this first

Conversation ID is the primary concept. Platform manages history automatically. Auto-bridges to Activity Protocol for Teams/M365 — no extra wiring.

Invocations

Workflow / app integration

Session ID is the primary concept. You manage state in your own code. Best for programmatic integrations and non-conversational workflows.

Activity

Teams / M365 channels

Used for Teams and Microsoft 365 channel integration. If you use OpenResponses, this is handled automatically on publish. No direct wiring needed.

A2A (preview)

Agent-to-agent delegation

Enables agent orchestration — one hosted agent delegating tasks to another. All four protocols can coexist in a single agent deployment.

03

Three types of agent

Foundry Agent Service supports three distinct agent types. Hosted Agents is the new capability in this release — the other two existed before.

GA ⚙️

Prompt Agents

Defined through configuration — instructions, model selection, and tools. No code required. Create them in the Foundry portal or via API/SDK.

Best for Rapid prototyping, internal tools, agents that don't need custom orchestration logic.
Preview 🔀

Workflow Agents

Orchestrate sequences of actions or coordinate multiple agents using declarative definitions. Build visually in the portal or define in YAML via VS Code.

Best for Multi-step orchestration, approval workflows, human-in-the-loop steps, repeatable automation without custom code.
New Preview 📦

Hosted Agents

Your code, packaged as a container. Any framework. Foundry manages the runtime, scaling, and infrastructure.

Best for Complex workflows, custom tool integrations, multi-agent systems, and scenarios where you need full control over agent behaviour.
Decision rule: Start with a Prompt Agent if you just need tools + instructions. Move to Workflow if you need visual orchestration or human-in-the-loop. Only reach for Hosted Agents when you need custom code, custom frameworks, or full runtime control. It's more powerful and more responsibility.

Prerequisites checklist

  • ✓
    Microsoft Foundry project — you need an active Foundry project in a supported region (East US, West US, UK South, Sweden Central, and ~20 others).
  • ✓
    Azure Container Registry — Foundry pulls your container image from ACR. Set up ACR pull access and RBAC before deploying.
  • ✓
    Docker / containerisation basics — you're packaging your code as a linux/amd64 container image. If you're new to Docker, this is the prerequisite skill to build first.
  • ✓
    Azure Developer CLI (azd) — the deployment tool. Install it and authenticate before trying to deploy.
  • ✓
    No secrets in images — use managed identities and Key Vault connections. Never put credentials in Dockerfiles or environment variables.
Post-publish RBAC: When you publish an agent to Teams/M365, Foundry provisions a new, distinct agent identity. The project managed identity permissions don't transfer. You'll need to reconfigure RBAC for every Azure resource your agent accesses after publishing. This is a common gotcha.

Pricing

Hosted Agents bill at $0.0994 per vCPU-hour of active compute. Idle agents scale to zero — you pay nothing. Sessions persist for up to 30 days; idle compute is deprovisioned after 15 minutes and seamlessly restored on the next request with the filesystem intact.

05

When to use it (and when not to)

Hosted Agents isn't the right answer for everything. It's the right answer when you need full control over agent runtime behaviour and want Microsoft to handle the infrastructure. Here's the practical decision framework:

✅ Use Hosted Agents when…

  • You have existing agent code in LangGraph, MAF, or custom Python/C# you want to run in production
  • You need custom tool integrations beyond what prompt agents support
  • Your agent writes files, executes code, or holds per-session context
  • You need multi-agent orchestration (A2A protocol)
  • You want one-click Teams/M365 publish with managed identity
  • Governance, auditability, and enterprise compliance are requirements
  • You've outgrown Copilot Studio's lifecycle ownership model

⚠️ Don't use Hosted Agents when…

  • A Prompt Agent (instructions + tools) is sufficient — don't over-engineer
  • Business users need to own and edit the agent lifecycle — use Copilot Studio
  • You need sub-100ms latency — in-process SDK agents are faster
  • You're prototyping and haven't validated the agent logic yet
  • Your team doesn't have containerisation skills yet — address the prerequisite first
  • Private VNet is a hard requirement today — BYO VNet for image pull is still in progress

Where this fits in the stack

What to do now

🔍

Audit your Ignite-preview agents

If your team ran any hosted agents from the Ignite 2025 preview, inventory them. This is a cutover, not an upgrade — plan a migration sprint.

📋

Identify your first real use case

Pick something small with clear success criteria. The EBR pipeline, the RFP builder, or an internal support agent are all candidates. Validate the container approach on one before scaling.

🛡️

Design identity upfront

Decide which Azure resources your agent needs to access. Design RBAC before you build, not after. Remember to redo it post-publish.

🚢

Don't skip the lifecycle check

Build version lifecycle polling into any automation from day one. Most delays come from treating "create version" as "deployed". Check for active before invoking.

🔗

Plan the Teams publish path

If the end goal is Teams or M365 Copilot delivery, test the one-click publish flow early. The RBAC reconfiguration is a known gotcha — don't discover it in production.

📚

Start with Microsoft Agent Framework

If you don't have strong opinions on framework yet, MAF has the deepest first-class integration with hosted agents and the most comprehensive quickstart path.
