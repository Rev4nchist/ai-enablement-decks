04 · Current state, honestly

The gaps — why it doesn’t flow yet

These are the present-day failures, stated as ground truth. They are the reason this document exists; the proposal below is the fix.

  1. No shared spine. The team dashboard reads the Tracker; the sponsor site reads the Cockpit instead — and the scheduled FourthOS run never logs to the Tracker. The two outputs don’t feed each other or share an upstream.
  2. Eve & Donna can’t reach the Tracker. Source = Eve / Source = Donna exist in the schema, but neither surface has a path that writes a Tracker row. Their work lands in Bridge queues only.
  3. Capture is manual / un-enforced. /log-work and the Bridge auto-mirror both depend on the model remembering to run them; no hook enforces capture. Unreported work is invisible to reporting.
  4. The team-dashboard pipeline is currently broken. The ai-reporting MCP server (which powers collect_weekly_work_tracker) is unregistered in settings.json / .mcp.json — the repo exists at C:/Users/david.hayes/ai-reporting-mcp/ but isn’t wired in, so /weekly-report Phase 1 will fail until it’s registered.
  5. The collector lives outside the repo. tracker_collector.py / the ai-reporting MCP are a sibling project (ai-reporting-mcp/), not in continuous-claude — easy to lose track of.
05 · The fix

One spine, two cuts

Keep two distinct output cuts (internal team vs VP/CTO sponsor — the audiences and content bar are intentionally different), but make the Weekly Work Tracker the single canonical capture spine that both cuts draw from. Sequenced, cheapest-first.

  1. Re-wire the team pipeline. Register the ai-reporting MCP (local stdio or Railway HTTP) and confirm /weekly-report Phase 1 runs. Without this, the team cut is dead. (External repo action.)
  2. Make the Tracker the spine. Every reportable unit of work lands exactly one Tracker row, Source-tagged, regardless of surface — the one rule the whole system hangs on.
  3. Close the capture gaps:
    • Code → the new reporting discipline skill (when/what/how to log; one-spine rule).
    • Donna → the Donna addendum (logs her Fourth work as Source=Donna rows).
    • FourthOS scheduled run → add a step that logs a Tracker row (Source=Code, Dashboard Section=Highlight) so the sponsor work is also visible to the team report.
  4. Two cuts read one spine. Team dashboard = full internal cut (unchanged). Sponsor site = filtered status-grade cut; optionally also read the Tracker for the cross-surface “what shipped” signal while keeping the Cockpit for portfolio framing.
  5. (Optional) light enforcement. A session-end nudge reminding Code to log unreported shippable work — a reminder, not a block — mirroring existing CCv3 hook patterns.

The principle: capture once, at the moment of work, into one tagged spine; render many cuts from it. Reporting quality is a capture problem, not a rendering problem — the renderers are already good.
