02 · Anatomy

The registry, stage by stage

Uniform by design. Every pipeline follows the same emit → upsert path, and the registry has a single writer — upsert.mjs. All of it is ESM with no external deps, living in scripts/report-registry/ and reusing scripts/project-cards/lib/notion.mjs.

Stage 1 — Emit: the report-run.json contract

make-run.mjs (or buildRun + writeRun imported directly, node-native) builds a flat, validated report-run.json and writes it to $TEMP/report-run-<source>.json. config.mjs is the single source of truth — it loads the canonical Notion IDs and enums from report-runs.ids.json. Contract keys: runId, type, period, runDate, status, artifactUrl, docxUrl, summary, source, commit.

Stage 2 — Upsert: the shared spine

upsert.mjs is the spine writer. It queries the data source for a row whose Run ID equals runId, then does update-in-place (same runId) or create/append (distinct runId). runId = <type>|<period>|<ISO-timestamp>, where the timestamp is the run date — so re-emitting with the same run date re-targets the same row (idempotent retry), and a fresh run date appends. On any failure it logs [report-registry] ERROR: and exits nonzero; every pipeline wraps the call so a registry outage can never redden the parent report.

04 · The arc

What the registry resolved

This document used to be a list of failures. Here is that same list — struck through — with what the Report Runs registry now does instead. The value is in the arc.

  • No shared spine. The two outputs didn’t feed each other or share an upstream; run state was siloed per pipeline.→ Resolved. All six pipelines now land one row in a single Report Runs DB (4e4c9460, ds c7d2d9e3), keyed by a unique Run ID — what ran, when, and whether it landed is queryable in one place.
  • The team-dashboard pipeline was broken. The ai-reporting MCP that powers collect_weekly_work_tracker was unregistered, so /weekly-report Phase 1 failed.→ Resolved. The ai-reporting MCP is registered in ~/.mcp.json (stdio, uv run … server.py, NOTION_TOKEN) — its collector + report tools are live.
  • The AI narrator had no history to read. There was no single place to answer “what ran last week / did it land?”→ Resolved. History is one deterministic query: ntn datasources query c7d2d9e3… filtered by Report Type — and append-all-attempts means every attempt, including failures, is its own row.
  • Historical backlog was lost. Past reports lived only as scattered on-disk artifacts.→ Resolved. backfill.mjs seeded ~46 real permanent history rows from VP archives, sponsor decks, the Self-Improvement INDEX, and Health-Check JSONs — deterministic Run IDs, so re-runs update in place and never duplicate.
  • Silent scheduled-report death. A pipeline could stop landing rows and no one would notice.→ Resolved. check-drift.mjs (read-only; exits nonzero on any type stale beyond 48h or with zero rows) is wired into the weekly Health-Check.
  • Retry-duplication & outcome-fidelity holes; Dashboard-Sync also swallowed its exit code.→ Resolved in the review pass (commit 9f689a4): idempotent create-or-adopt on transient failure, chronological (not lexicographic) newest-row selection, URL-typed-property guard, Health-Check worse-state default, Source-enum validation, orphan-duplicate cleanup, and exit $LASTEXITCODE.
  • A registry outage could block a report.→ Resolved. upsert.mjs is non-fatal-but-loud and every caller wraps it — a Notion hiccup degrades to a missing row (caught by drift), never a failed parent run.

The principle held: capture once at the moment of the run, into one tagged spine; render many cuts from it. Reporting observability was a capture problem, not a rendering problem — and capture is now uniform across all six pipelines.

05 · Current state, honestly

What still isn’t wired

The spine is real and live, but it is not finished. These are the present-day open items, stated as ground truth — several are deliberate deferrals (see tasks/tasks-report-registry.md “Not in scope”).

  1. No live per-run preview embeds. The Phase 3 hybrid “Current run” preview cards are explicitly deferred (MCP-only, size-gated). The registry today is history rows + linked views, not live per-run preview cards.
  2. The consumers live out-of-repo. The ai-reporting MCP server (C:/Users/david.hayes/ai-reporting-mcp) and the dashboard (Projects/ai-reporting-dashboard) are not version-controlled inside continuous-claude. The spine’s contract (README.md schema) is the only in-repo coupling — the consumers can drift independently.
  3. Two report types start with empty history. Backfill seeds only 4 of 6 types; Team Dashboard and Project Portfolio have no historical on-disk artifacts, so they fill only from the next live run — and may show as drift (zero rows) in check-drift until then.
  4. The hub launcher isn’t wired. The hub “📥 Latest reports” launcher rewrite is additive-only and deferred — flagged as human-territory, not machine-owned. The hub’s top section is not yet bound to the registry.
  5. The 7 views have a cosmetic placement bug. notion-create-view appended each linked view at page-bottom rather than nested under its ## History heading. Filter and sort are correct — only the visual placement is off.
  6. Accepted (unguarded) races. Two overlapping runs of the same source share the canonical $TEMP/report-run-<source>.json emit path and can create/create-race on the same runId. Only Task Scheduler’s no-overlap (MultipleInstances=IgnoreNew) prevents it; the >1-match orphan-cleanup is the self-heal if it ever fires.
  7. A registry row is best-effort. Because upsert is non-fatal, a transient Notion failure that even the create-or-adopt retry can’t recover silently drops that run’s row until the drift detector notices — there is no synchronous guarantee every run is recorded.
