Authentication architecture & the reusable SaaS-connector pattern

How we bridge Microsoft Entra ID and Salesforce in a FastMCP governance proxy — and what carries over to Zendesk, NetSuite, the Product MCP, and other downstream SaaS connectors.

Author: David Hayes 2026-05-14 Edge auth: done Downstream: 3/4 green — audience fix pending

TL;DR

The proxy bridges two identity worlds with two independent OAuth legs.

  1. Edge auth — incoming MCP requests are validated by ACA Easy Auth (Entra-backed) and re-checked by an in-process middleware (EasyAuthGate). Defense in depth.
  2. Downstream auth — outgoing calls to Salesforce use OAuth 2.0 Token Exchange (RFC 8693) so the calling user's identity flows through. No shared service account.

Both legs are designed to be reusable for other SaaS connectors — Zendesk, NetSuite, the Product MCP, or any third-party we put behind an Entra-rooted governance proxy. The edge middleware is SaaS-agnostic; the downstream pattern adapts to whatever OAuth server the target system runs.

1The two-leg picture

Every request crosses two identity boundaries. We have to validate the inbound Entra token, then trade it for a Salesforce token under the same user. No single token satisfies both sides.

2Leg 1 — Edge auth (client → proxy)

Only Fourth-approved users carrying the Mcp.User App Role can reach any tool on this proxy. Easy Auth at the platform edge does the JWT validation; EasyAuthGate in our process does the role check.

Key implementation detail

Easy Auth is configured with Return401, not RedirectToLoginPage. MCP clients can't follow HTML redirects; they need a clean 401 to initiate the OAuth dance properly. This is one of the most common deployment mistakes for MCP-on-ACA.

The role-claim matcher in EasyAuthGate accepts both roles and the schema-URI form (http://schemas.microsoft.com/ws/2008/06/identity/claims/role) so it works with both Entra v1 and v2 tokens.

3Leg 2 — Downstream auth (proxy → Salesforce)

Every upstream call carries the calling user's identity, not a shared service account. The bridge is OAuth 2.0 Token Exchange (RFC 8693) — the cleanest way to swap one IdP's token for another's without redirecting the user through a second sign-in.

Why Token Exchange and not PKCE-only?

PKCE works, but it requires a user-side redirect through Salesforce on first auth (and on token expiry if there's no refresh token). Token Exchange keeps the user inside the Microsoft sign-in flow — they never see Salesforce's login page. PKCE remains the documented fallback if the Salesforce-side handler proves operationally painful.

4Token storage

Short-lived access tokens stay in memory. Long-lived refresh tokens eventually move to Azure Key Vault — one secret per user, referenced via managed identity to keep them out of the control plane.

5Reusability for other SaaS connectors

Scope note. This section is about other MCP servers fronting downstream SaaS systems — Zendesk, NetSuite, the Product MCP Server, any third-party we want to broker behind an Entra-rooted governance proxy. It is not about exposing Fourth's own product line via MCP; the two have nothing to do with each other.

Short answer: yes, most of the auth design transfers directly. The edge leg is fully SaaS-agnostic — every Fourth MCP connector behind Easy Auth gets EasyAuthGate for free. The downstream leg is a pattern (Entra-rooted identity → exchange or OBO → short-lived downstream token), not a Salesforce-specific implementation.

Copy as-is

Edge middleware

salesforce_mcp/auth.py (EasyAuthGate) only depends on X-MS-CLIENT-PRINCIPAL. Pass your own role name (e.g. Mcp.Product.User) and you're done.

Copy as-is

Token store

token_store.py — TokenStore Protocol, CachedToken, get_or_acquire. Pure caching, not Salesforce-aware. Phase-7 Key Vault swap is a one-line factory change.

Copy as-is

Config shape

config.py — the dataclass shape (Entra block + downstream block + auth-mode switch + transport block) generalizes. Rename the SF fields to whatever the Product system needs.

Adapt

Token exchange

exchange_entra_for_salesforce_token() is named for Salesforce and posts to /services/oauth2/token. The RFC 8693 flow is reusable — just retarget the URL and body fields. If the Product system doesn't speak RFC 8693, fall back to OBO (if it's Entra-native) or PKCE.

Adapt

Hosted MCP client

client.py assumes Salesforce's /services/mcp/<server-name> path. Swap the URL builder but keep the per-request token-acquire + JSON-RPC envelope pattern. The whole shape is ~100 lines.

Adapt

App registration

Register a separate Entra app per connector. Distinct App ID, distinct App Role (Mcp.Zendesk.User, Mcp.Netsuite.User, etc.), distinct Custom Connector on Anthropic. Independent role assignments, independent secrets, clean audit trail.

Do not copy

Salesforce ECA specifics

PKCE S256, Token Exchange Flow, Apex Auth.TokenExchangeHandler — Salesforce-flavored. Only relevant for Salesforce itself. Each SaaS has its own OAuth implementation and handler equivalent.

Do not copy

Schema discovery

Salesforce-specific. Each SaaS has its own schema model. For Salesforce we're now planning to consume Ivan's Apex package — it maintains a Schema_Snapshot__c custom object refreshed hourly via Queueable chain inside the org, so we just SOQL the latest Run_ID__c through the proxy instead of walking /services/data/v62.0/sobjects ourselves. Governor limits solved in-org.

Do not copy

Tool profiles

marketing-readonly and friends curate against Salesforce's 60+ Hosted MCP ops. Each connector curates against its own surface area; reuse the pattern (allowlist middleware), not the specific profiles.

Recommended starting point for any new SaaS connector

  1. Read the ~/continuous-claude/docs/fastmcp-connector-playbook.md end-to-end. Canonical recipe at Fourth; supersedes ad-hoc tribal knowledge.
  2. Copy salesforce_mcp/auth.py and salesforce_mcp/config.py into your new package (e.g. zendesk_mcp/, netsuite_mcp/, product_mcp/) as the scaffold.
  3. Register a separate Entra app + App Role + Anthropic Custom Connector per connector.
  4. Decide the downstream auth mechanism based on what the target SaaS's OAuth server supports (RFC 8693 / OBO / PKCE).
  5. Reuse the Cosmos audit/rate-limit middleware classes — none are coupled to Salesforce.

Should be a half-day exercise given how much of the chassis is already built. Happy to pair on the first one.

7Open questions worth flagging

  1. Token Exchange Handler audience config — handler is registered (✅), but configured for the App ID URI form (api://<guid>). v2 Entra tokens carry the bare GUID as aud, so a one-line config update is needed on the SF side. No code redeploy.
  2. OBO confirmation against the SF ECA — Phase 0 spike uses client_credentials for the Entra leg, not OBO. Need to validate OBO works before Phase 3 ships true per-user impersonation.
  3. Per-user refresh-token storage in Key Vault — needs IT/security signoff. Not blocking Phase 1/2 since we're cache-only for now.
  4. Anthropic Custom Connector + DCR — confirm Fourth's Entra app is configured to allow Dynamic Client Registration (should be by default — App Role-gated, not DCR-blocked).
