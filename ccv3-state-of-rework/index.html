CCv3 State of Rework

Clean spine, baroque edges: 190 findings, three dead safety chains, one write-dead bus, a 22-32s hot-path tax — all rewire-in-place on a sound core.
What it is

CCv3 is a self-improving Claude Code substrate: six integrated pillars (Memory / Hooks / Agents / PageIndex / Workflows / Braintrust) on a WS-2 code-intelligence substrate (codegraph L3 specialist → /code-intel facade L4 → L2 context bus) that compounds a Claude instance's effectiveness across sessions via persistent BGE-vector semantic memory, TypeScript lifecycle-event hooks compiled to ESM, specialized sub-agent delegation via the Task tool, Ralph/Maestro workflow orchestration, and Braintrust telemetry with LLM judges.

Why we're honing it

The Fable-5 deep review (WF-0 harvest → WF-1 discovery → WF-2 WF-3 verify+synthesis at SHA 86b8f60) surfaced 190 confirmed findings (S0:3, S1:41, S2:108, S3:38) plus a Codex gpt-5.5@xhigh cross-model pass (184 independently corroborated, 6 codex-only additional findings, 2 S1→S2 Codex downgrades). Elegance verdict: mixed. The SQL layer, secret handling, path-traversal defenses, and L3 boundary/bus design are cleared by the D5b coverage manifest; the enforcement and intelligence periphery is largely dead or mis-wired. Three structural seams are active today: (1) the entire agent safety+verification chain (12 hooks) never fires because all are registered under matcher 'Agent' while the live Claude Code tool emits 'Task'; (2) the context bus symbol layer is write-dead because SubagentStop has zero registrations anywhere; (3) seven competing getSessionId implementations produce non-joinable telemetry, causing 60.7% corr-null on intel-bus rows. The dominant failure pattern is registration/wiring drift plus consolidatable duplication — prune-and-rewire on a sound core, not fundamental over-design.

Where it lands

The elegant end-state: a single-writer context bus where SubagentStop commits sub-agent proposed_bus_updates so cross-agent context is never silently dropped; a lean UserPromptSubmit hot-path with a resident recall daemon eliminating the 22-32s 13-spawn serial wall; one canonical getSessionId, one storeLearning(argv) spawnSync helper across all call sites, one injectRecall() sanitizer before any archival content reaches model context; behavior evidence-locked via §9 facade-bypass counts in intel-bus.jsonl so the C.5 deny-flip gate triggers only after ≥80% adoption is measured in actual telemetry; WRRF source weights recalibrated from 30 days of bus-wrrf.jsonl rather than held as educated guesses; and no always-loaded doc surface asserting enforcement that has never fired.

Data-Flow Reality — Traced End-to-End Paths, Seams, Races

Four traces grounded in live .claude/settings.json registration order and .claude/hooks/src/*.ts source logic at HEAD f7f3eba.

Trace 1: UserPromptSubmit → memory recall → bus read → context injection

12-step serialized chain: heartbeat (Postgres sessions heartbeat) → maestro-state-manager → guardrail-enforcer → skill-activation-prompt → bus-session-populator [WRITE — must precede step 6; ordering load-bearing for WS-2 B.3b/B.4a; mutateBus(bumpTurn + setIntent), 120-char sanitized prompt cap] → memory-awareness [readBus focus_symbols → isDaemonReady TCP probe → hybrid RRF or text-only FTS mode → parallel checkLocalMemory + checkDbMemory → mergeResults → applyFloor → Braintrust emit → MEMORY MATCH + SESSION FOCUS inject as additionalContext] → maestro-detector → user-confirmation-detector [RESOLUTION_SIGNALS → storeVictoryFromConfirmation via spawnSync argv post-QW-01] → ralph-watchdog → ralph-progress-inject → ralph-retry-reminder → braintrust_hooks.py (root span open/continue).
